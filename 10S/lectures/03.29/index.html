This should not be your first programming course!

You should already have had a course where you learned the following, perhaps in Python, Java, Visual Basic, C, C++, C# or some other programming language.

If you took CS8 and got a C-, D+, D, D- or F:

  • then: CS16 may not be a good choice for you—please repeat CS8 first.

You should be comfortable with at least the following.

  • variables, expressions. assignment statements (e.g. x = x + 1 makes x one more than it used to be)
  • order of operations (operator precedence), e.g. 2 + 4 * 5 gives 22, not 40
  • if/else, such as: 
    if (x<0)
      printf("Negative);
    else
      printf("Positive);
  • loops (while loops, for loops)—for example, if x starts at 16, what value will count be when this loop is finished—when we reach the line return count; ? 
    count = 0;
    while (x>1)
{
      x = x / 2;
      count = count + 1;
    }
    return count;
  • functions (function definition, function call, arguments, parameters)
    • Why do we use functions in a program—what is the problem that they solve?
    • (If you studied Java: you may be familiar with the word method rather than function. For purposes of this discussion—if you know what a method is, then you know what a function is. The difference between method and function comes up in later courses (CS24, CS32), but isn't important for CS16.)
  • arrays
    • If you studied Python, you may know these as lists
    • a[0], a[1]. a[2], etc...

Since this is not a first programming course, we'll move quickly

  • We don't expect you to already know C
  • But we do expect you to already know how to program
  • So, we'll be:
    • moving through chapters in the book quickly
    • sometimes introducing a few details of C earlier than they appear in the book

Three topics we want to introduce early:

  • arrays
  • structs

We want to try a different approach:

Basics of using Ch

There is a very useful program on CSIL called Ch that you can use to learn about some of the basic properties of the C programming language.

Ch is an 'interpreter' for C—it allows you to type in C statements one at a time, and have those statements be immediately turned into something that can be executed by the computer, one at a time. This is really helpful for trying out small bits of code to see what they will do.

Ch has many other uses, as we may see later in the course—but for today we'll primarily be using it for little experiments to learn about:

  • declaring variables
  • assignments statements
  • the printf command.

To bring up Ch on CSIL, login in, then simply type ch at the command prompt.

You should see something like this (except the directory will be different) 

-bash-3.2$ ch

                                   Ch 
                Professional edition, version 6.1.0.13751 
              (C) Copyright 2001-2009 SoftIntegration, Inc.
                     http://www.softintegration.com

/cs/faculty/pconrad>

The /cs/faculty/pconrad that appears in the prompt is an indication of what directory you are in—for you it, may be something such as:

/cs/student/bettywhite>

or

/cs/guest/avigoda>

Customizing the Ch prompt

When working in Ch, I typically change the prompt to be the letters Ch> and I invite my students to do the same.

  • Changing the prompt like this helps to remind us that we are working in Ch
  • If we both do it, then the examples of Ch sessions I provide to you will look the same as what you see on your screen.

Here's how: while in Ch, type the following command: _prompt = "Ch> "

Here's what doing that may look like:

/cs/faculty/pconrad> _prompt = "Ch> "
Ch>

Note the following important details:

  • the underscore _ at the beginning of _prompt
  • the space after the > and before the " inside "Ch> "

Now, your prompt is Ch> (with a space after it).

Getting out of Ch

To get out of Ch and get back to the -bash-3.2$ prompt, just type exit like this:

Ch> exit
-bash-3.2$

Now, let's get back into Ch, and explore declaring variables in C:

Backgroud about text editors:
"Why you should emacs (or vi, but not pico or nano)"

Typing in code to ch one line at a time is great for learning some of the basics of C and getting immediate feedback—but when we work with longer programs, we need a way to save them in a file.

A program that will allow us to create and modify files on the disk is called an editor.

The editor we'll concentrate on in this course and recommend to you is: emacs

  • emacs is what we recommend if you don't already know an editor
    • It offers a balance between being powerful, and being easy to learn.
  • vi or vim is also a good choice
    • But only if you already know it—in my experience, it is harder for beginners to learn.
  • You are free to use any editor you like, but we recommend against pico or nano—more below.
  • We also recommend against transferring files to your PC and using Notepad (Windows) or TextEdit (Mac)
    • You need to learn an editor on CSIL so that you can be effective team member

 

Text editors vs. IDEs

You may have worked with a "integrated development environment (IDE)" before such as:

  • IDLE for Python
  • Visual Studio for .net, Visual Basic, Visual C++, C#, etc.
  • BlueJ, Netbeans or Eclipse for Java

Those environments are very useful tools—and in a later course (CS32) you'l work with one called Netbeans that has great features for working with large projects in C++ and Java.

However, it is also very handy to know how to use a plain text editor—and CS16, that's the environment we'll emphasize:

  • Knowing how to work with programs in a plain-text editor is a basic skill that many employers will expect you to have.
  • It is very useful in a wide variety of real-world situations—e.g. when the fancy tools are not working properly, and you simplify and get down to what is "under the hood"

emacs vs. vi—a "religious" war?

There are two "plain editors" that are widely used in the software development world: emacs and vi

  • Note that each comes in flavors: emacs may be also be xemacs, vi may also be vim
  • The folklore among software developers is that there is a sort of "religious war" between vi and emacs
    • See for example, this cartoon: http://xkcd.com/378/
  • A few of the differences
    • vi tends to be smaller, requires less disk space and memory, and load faster
    • emacs tends to be easier for beginners to learn—vi modes can be frustrating for beginners
  • Both offer:
    • syntax highlighting, special features for programmer productivity (multple buffers), an empahsis on code
  • Why nano and pico are discouraged
    • pico was originally designed for editing text (as part of the pine email system) and lacks features needed for programmer productivity
    • nano does incorporate some of those features, but not enough for us to recommend it

Once you learn an editor, it is difficult to retrain your fingers for another one—so choose a good one.

Emacs basics

About the control characters:

  • C-x is read as "control C" and it means: hold down control, and type x
  • C-x C-c means "control X control C" and it means: hold down control, and type x then type c
  • etc.
To do this Here's what you type:
Start emacs from the bash command line
to create or modify the file hello.c		 At the bash prompt, type: emacs hello.c.

If hello.c doesn't already exist, you'll get an empty window to start typing.

If it does already exist, the file will come up so you can make changes.
Leaving the editor (with or without saving changes)

C-x C-c

If you have made changes, you'll be asked to save:

  • type y to save changes
  • type n to leave without saving changes (see note*)

 
Moving around arrow keys: ←  ↑  →  ↓ 
Deleting the previous character delete key: ⌫
Deleting the current character C-d
Getting out of trouble if something weird happens C-g
Saving the file without leaving emacs C-x C-s

*Note: If you are asked if you want to save changes and you hit n for "no", as a safety measure emacs will ask:
Modified buffers exist; exit anyway? (yes or no)

In this case you must spell out the word yes to indicate that you don't want the changes saved.

More on learning emacs is available here: http://www.cs.ucsb.edu/~pconrad/topics/emacs/
