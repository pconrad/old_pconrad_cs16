Errors #1-#6: the six hints

I used error code #1 through #6 for the six hints that were listed right in the question text:

Hints for full credit
  • #1: Use correct % specifiers (%d or %i for int, and %lf for double)
  • #2: Remember the special syntax detail that applies to using scanf.
  • #3: Remember the line of code to access the standard input/output library
  • #4: Use "function prototypes" when function calls precede function definitions
  • #5: Do what is needed at the end of the output
    so that the Unix prompt after the program ends appears on a new line
  • #6: When the main() is finished, remember to signal success/failure to the OS

Error #1: (-1)

No one made this error: everyone that used scanf used the correct specifiers, either %d or %i

scanf("%d",&width);

Error #2: (-1)

This one referred to using & before the variables in scanf:

scanf("%d",&width);

Three papers omitted the & completely, e.g. scanf("%d",width);
One papers use a @ instead, i.e. scanf("%d",@width);

Be sure you learn to draw the & symbol—it's a skill you'll need for later C/C++ courses.

Error #3: (-1)

This one referred to remembering to insert this line before the main

#include <stdio.h>

Error #4: (-2)

Because the function starBox is called (inside main) before it is defined, it is necessary to have a function prototype before the main:

void starBox(int width, int height);

This should go above the main program. One person put it inside the main program, for which I only took off 1 point for style. Strictly speaking, it is legal to put function prototypes inside the main, but it is not good style to do so.

Error #5: (-2)

The last thing printed by a program should be a \n character.

It turns out that the starBox function already takes care of this, so no-one lost points on this item.

Error #6: (-2)

At the end of the main function, you need this line to signal success to the OS

return 0;

Errors #6-#14: other errors

Error #7: putting types before params in a function call (-2)

This error pertains to the correct way to make a function call. In a function call, we do NOT precede the names of the actual parameters with their types. This is ONLY done in a function prototype, and the first line of a function definition.

Incorrect Correct 
starBox(int width, int height);
 
starBox(width, height);

The reasons is that width and height are NOT being declared here as brand new variables. We only precede a variable with its type when it is being declared—i.e. introduced for the first time. (An exception to the 'first time' rule is that we repeat the type between the function prototype and the function definition.)

Remember: no types before parameters in a function call!

The only real problem with the offered answer below is a missing comma between &width and &height. Strangely enough, the lack of a comma between "%i" and "%i" is actually correct—because in C, if you have consecutive string literals with no punctuation in between (only white space), then the string literals are treated as if they were concatenated.

Error #10: extra \n in scanf statement (-1)

In a scanf statement, many students will common place an extra \n in the format specifier. Although some textbooks suggest this, in practice, it frequently causes problems, so I advise against it, and teach not to do it. As a reminder not to do this, I deducted one point on papers where this appeared in the answer:

Example:

Incorrect Correct 
scanf("%i\n",&width);  
scanf("%i",&width);

Error #11: forgetting to declare local variables (-2)

In C, local variables inside main, or any function or block for that matter, must be declared at the top of that function (or block), just after the opening curly brace ({).

Example:

Incorrect Correct 
...
int main()
{
   printf("Please enter width:");
   scanf("%d",&width);
   ...
 
...
int main()
{
   int width,height;
   printf("Please enter width:");
   scanf("%d",&width);
   ...

These should be declared local to the function in which they are used, and not global (see error @@ below).

It is not necessary—and in fact, is an error—to use atoi to conver the result of doing a scanf that reads data in with %i or %d. If data is read in with %i or %d, then it has already been converted to an int.

Error #14: width and height should be local to main, not global; (-2)

Global variables are almost always a bad programming practice in C—there may a very small number of cases in future courses where they are warranted, but the general rule is that they are to be avoided—especially in first year programming courses.

If you use a global variable in your C/C++ programming, you should have a very good reason to do so—and in this course, there will NEVER be such a reason. If you think you need a global variable, it is a sign that you are misunderstanding the problem in some way.
