Minor error 2: ^2 doesn't do exponentiation (-2) (2/72 papers)

 

In some languages, the ^ operator can be used to raise a quantity to a power. In C/C++, however, the ^ operator doesn't raise a quantity to a power. Instead, it performs the operation "bitwise exclusive or", which is an entirely different operation.

That's an bit of C's history as an OS development language showing through: the bitwise exclusive or operation is useful in low-level programming for things like device drivers, network routers, compression algorithms, etc., but not so much in a first programming course. So, this is one of those things you just need to memorize: stay away from ^ in C unless you know what "bitwise exclusive or" is, and know that this is what you really want!

Incorrect Correct 

int squareOfDiff(int a, int b)
{
  return (a-b)^2;
}
 

int squareOfDiff(int a, int b)
{
  return (a-b) * (a-b);
}

Minor error 3: void vs. int for return type (-2) (6/72 papers)

If a function contains a return statement with something after it, such as (a-b)*(a-b), it cannot be a void function. Instead, the return type must match the return type of the expression that comes after the word return, i.e. in this case, int.

Incorrect Correct 

void squareOfDiff(int a, int b)
{
  return (a-b) * (a-b);
}
 

int squareOfDiff(int a, int b)
{
  return (a-b) * (a-b);
}

Minor error 4: using Python keyword def in C function definition (-2) (1/72 papers)

In Python, function definitions start with def. But in C, they start with the type of the quantity returned, e.g . int, double, etc. or void if the function doesn't give back a return value.

(Side note: In Python, a given function can return more than one type of expression, but in C, any given function is restricted to only one type of return value. There are pros/cons to each approach to language design—that's something you can learn more about in the upper division course CMPSC162.)

Incorrect Correct 

def squareOfDiff(int a, int b)
{
  return (a-b) * (a-b);
}
 

int squareOfDiff(int a, int b)
{
  return (a-b) * (a-b);
}

Minor error 5: omitting return type in C function definition (-2) (5/72 papers)

In a C function definition, the name of the function is preceeded with the type of the quantity returned, e.g . int, double, etc. or void if the function doesn't give back a return value.

Strictly speaking, it is permitted to omit the return type if the return type is int. However, I would still argue that to rely on this is a serious style violation, worth a 2 point deduction.

Incorrect Correct 

squareOfDiff(int a, int b)
{
  return (a-b) * (a-b);
}
 

int squareOfDiff(int a, int b)
{
  return (a-b) * (a-b);
}

Minor error 6: pow not appropriate in this context (-1) (1/72 papers)

There is a built-in function called pow that will raise a quantity to a power. However, the value it returns is of type double. So, if it were used here, you'd either need to convert the result back to an int, or have double as the return type of the function.

In general, pow is appropriate when working with doubles, but not when working with integer quantities.

This is a debatable point, but in the end I decided that a 1 point deduction was warranted, if nothing else, to call attention to the issue.

Incorrect Correct 

int squareOfDiff(int a, int b)
{
  return pow(a-b,2);
}

{
  return (a-b) * (a-b);
}

major error 1: misunderstanding printf vs. return (-2 or -5) (8/72 papers)

One of the most crucial things to understand in this course is the difference between printing a result on the screen, and returning a value from a function.

Here's a sample of an incorrect answer vs. a correct answer:

Incorrect Correct 

int squareOfDiff(int a, int b)
{
  int result = (a-b) * (a-b);
  printf("result=%d\n",result);
  return 0;
}
 

int squareOfDiff(int a, int b)
{
  int result = (a-b) * (a-b);
  return result;
}

The answer on the right—the correct answer—shows how to return a value.

Why does this distinction matter so much? Here's why:

When a function returns a value, that value can be used in many different ways: it can be printed on the screen, written to a file, sent over the internet, passed to another function, or stored in a variable for later use.

By contrast, a function that just prints a value can do one and only one thing—print that value. Period. This is not nearly as useful.

Please—if you made this mistake, and don't understand the difference, come to office hours for additional help with this issue.

A less serious, but still signifcant version of this error was including a gratuitous printf in an otherwise correct function. This only resulted in a deduction of -2, but is still a significant misstep:

Incorrect Correct 

int squareOfDiff(int a, int b)
{
  int result = (a-b) * (a-b);
  printf("result=%d\n",result);
  return result;
}
 

int squareOfDiff(int a, int b)

major error 3:
shadowing formal parameters (-5) (3/72 papers, 8%)

This is a subtle error, but an important one to learn about.

If you declare a local variable that has the same name as a formal parameter, this is called "shadowing" the parameter.

If you look up in the night sky, you may be able to see many stars. If you can't see the stars, it doesn't mean they are not there—it just means that there may be clouds or fog in the way. And if you look up in the day time, the stars are still there—you just can't see them because the sun's light is too bright.

What does this have to do with C programming? Consider the following pair of incorrect and correct functions:

Incorrect Correct 

int squareOfDiff(int a, int b)
{
  int a;
  int b;
  return (a-b) * (a-b);

}
 

int squareOfDiff(int a, int b)
{
  return (a-b) * (a-b);
}

The functions are the same, except the incorrect one has two extra lines declaring local variables a and b. The problem is that when we compute the value result, instead of using the values that were passsed into the function, we are using the uninitialized values of the local variables a and b. These local variables happen to have the same names as the formal parameters a and b, but they are different variables.

Thus, the value returned will be completely unpredictable.

So, it is important to NOT declare local variables with the same names as your formal parameters.

This is a very common mistake that students make not only in C (in courses like CS16) but also later courses in C++ such as CS24, CS32, CS130A and in Java in CS56. This comes up again in writing member functions for C++ and Java classes (where not only can you shadow a formal parameter, but also a class variable). So it is good to learn about this idea early on.
