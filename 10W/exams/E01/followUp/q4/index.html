Question 4 was an essay question which asked you to explain the statement:

First, see the test fail
Then, see the test pass

The main idea here was the one explained in step 7 of lab02, which discussed a broken test harness.

Since this was an essay question, the grading was admittedly more subjective than with other questions on the exam.

Here are three examples of answers that were given full credit (10/10). I've added my own emphasis (bold type) to these answers to indicate the main point I was looking for. It was necessary for your answer to state this point clearly in order to get full credit.

  • Using stub vaues, which give expectd false values, we want to see if the test works. If the test is functional, all of the stub values will make the test fail. The programmer can then move on and remove the stubs, and hopefully the test cases will work.
  • The first programmer failed to understand the test-driven deveopment methodology. While having the code pass on the first try could indicate that the code is correct, it could also indicate that the test itself is flawed. Hence by verifying that the tests fail, one confirms that code that passes the tests is correct and not a byproduct of a faulty test.
  • When you write the code, you first write it incorrectly on purpose. This is a way to test the test and ensure that it won't pass if the code is wrong. Then you can fix the code so the test passes.

Other answers were awarded 9/10, 8/10, 7/10, 6/10, or 5/10 based on how close they were to a clear and cogent explanation of this point. Answers that received 8/10 or 9/10 may have hinted at the key idea without stating it clearly. Answers receiving 5,6 or 7/10 tended to be statements of things about coding and testing that were true, but that missed the main point. More points were awarded to answers that were more relevant and clear, while fewer points were awarded to vague and unclear answers.
